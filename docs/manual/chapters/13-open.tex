\chapter{Open Questions}

The narrow v1 contract --- one volume, one pinned state, namespace
plus logical size, fallback for everything else --- has been the
project's anchor through R1. R2 opened three named lanes against it.
This chapter is what each lane has resolved and what each has not,
written so a reader can see what the project knows and what it has
deliberately left unclaimed.

The chapter is organised by lane rather than by topic. A lane is a
unit of work whose entire scope --- source class, semantic mode,
oracle, support claim --- can be specified before the work starts.
Lanes are isolated from each other; finishing one does not entail
finishing any other.

\section{R2-A: Allocated Size --- Landed With Two Fail-Closed Cases}

The most visible WizTree feature R1 did not yet provide.

\textbf{The question.} For each inode, what number of bytes does
APFS allocate to its content on disk?

\textbf{The result.} Five of six case classes emit a per-inode
\texttt{allocated\_size} that matches macOS's
\texttt{st\_blocks * 512} oracle exactly. Two cases fail closed in
v1: sparse files (where \texttt{j\_dstream\_t.alloced\_size}
overstates the actual allocation by the sparse-bytes hint) and
decmpfs-compressed files (where no on-disk field has been
oracle-validated against the macOS number). Chapter~8 gives the
precedence rule, the case-class table, and the per-row evidence;
chapter~11 records that the visualisation greys the metric toggle
out for any row that lacks a value, and any directory aggregate
containing such a row propagates \texttt{None} upward.

The fail-closed cases are not bugs. They are scope choices: the
sparse hypothesis (\texttt{alloced\_size} $-$ \texttt{sparse\_bytes}
$=$ \texttt{st\_blocks * 512}) holds on the one fixture that
exercises it, but is one data point and needs a broader sparse
corpus before it can be promoted from Hypothesis to Observation.
The decmpfs gap is genuine: no public reader has been validated
for the resource-fork-backed decmpfs path on macOS, and v1 will
not emit a number it cannot defend.

\textbf{What did not land.} Exclusive bytes (bytes only this inode
owns, after accounting for clones and snapshots) and shared bytes
(bytes shared across inodes or snapshots) require the
extent-reference tree to be opened and joined against per-inode
extents. Neither is part of R2-A; both are separate lanes with
their own oracles.

\textbf{Future direction.} An EX-22b sparse-corpus probe (leading
hole, trailing hole, multi-hole, hole-larger-than-file, zero-hole
boundary) would close the sparse fail-closed case. The
\texttt{alloced\_size - sparse\_bytes} identity is the candidate;
the EX-22b oracle is the same \texttt{st\_blocks * 512} that R2-A
used.

\section{R2-B: Snapshot-Assisted Scanning --- Entitlement-Blocked}

The mechanism by which a fast indexer can scan a live machine
honestly.

\textbf{The question.} If the indexer takes a local APFS snapshot
of a live volume, mounts it read-only, and walks it through the
existing fallback, does the result match what the same walker
would have emitted on the live volume at the snapshot moment?

\textbf{What landed.} A source review of the snapshot API surface
(SR-020) made the privilege boundary explicit:
\texttt{fs\_snapshot\_create(2)} requires root \emph{and} the
\texttt{com.apple.developer.vfs.snapshot} entitlement; the same
gate applies to delete, rename, mount, and revert.
\texttt{mount\_apfs -s} also requires root.
\texttt{tmutil~localsnapshot} is the only unprivileged create
path; it invokes \texttt{backupd} over XPC and only works on
volumes included in the Time Machine backup set.
\texttt{fs\_snapshot\_list(2)} and the corresponding
\texttt{diskutil} and \texttt{tmutil list} subcommands are free.

The scanner therefore cannot stage its own snapshot inside an
unprivileged test process. The R2-B Rust integration --- a
\texttt{--snapshot <mountpoint>} flag that consumes an
already-mounted snapshot directory and runs the existing fallback
against it --- is operationally useful immediately. The
shape-parity claim itself is parked behind an unblock condition:
the EX-23 probe ran on a clean dev workstation, found no
user-visible snapshots (only the sealed-system
\texttt{com.apple.os.update-*} snapshot, which it correctly
skipped), and emitted a \texttt{blocked\_no\_snapshots\_at\_all}
verdict with the exact \texttt{sudo mount\_apfs -s} reproducer
command saved in the probe artefacts.

\textbf{What did not land.} The shape-parity oracle itself.
Until a privileged rerun produces a positive verdict, the
\texttt{--snapshot} flag tags the source as a snapshot in
\texttt{correctness\_claim} and \texttt{not\_claimed} but does
not promise that the result matches a live-directory walk on this
host. The \texttt{not\_claimed} register names the missing
oracle explicitly so a consumer cannot accidentally treat a
snapshot scan as equivalent to a live scan.

\textbf{Future direction.} A privileged rerun. The reproducer
is:

\begin{minted}{bash}
tmutil localsnapshot /
tmutil listlocalsnapshots /
sudo mkdir -p /tmp/apfsfi-ex23-snap
sudo mount_apfs -s <snapshot-name> / /tmp/apfsfi-ex23-snap
python3 docs/research/experiments/EX-23-snapshot-shape-parity/artifacts/probe_ex23.py
sudo umount /tmp/apfsfi-ex23-snap
\end{minted}

A positive verdict would unlock moving the snapshot-mount cell of
the support matrix from \texttt{not\_claimed} to
\texttt{fallback\_supported}.

\section{R2-C: Scanner Ergonomics --- Two Levers Investigated, One Shipped}

The lane that does not change semantics, only the surface.

\textbf{The questions.} Which parts of the scanner's
user-facing surface --- backend selection, progress reporting,
embeddability in a native UI, attribute fetch cost, parallelism
--- can be improved without moving the emission contract?

\textbf{The performance-lever map.} Source review SR-021
synthesised four parallel research threads (xnu source, the
Spotlight alternative, an OSS-tool survey, and the
\texttt{jwalk}/\texttt{rayon} prior art) into one map.

\paragraph{Falsified.} EX-24 measured the attribute-mask shrink
lever: drec-only attributes (no inode-required fields) versus the
production walker's attribute mask
(\texttt{ATTR\_FILE\_TOTALSIZE | ATTR\_FILE\_ALLOCSIZE}).
The two rows landed within one per cent of each other. The
SR-021 hypothesis that the inode-attribute fetch path crosses
xnu's vnode-rage boundary as a measurable lever is
\emph{falsified for modern Apple silicon and this attribute set}.
The Phase-2 deferred-attribute refactor is therefore
de-prioritised. The \texttt{fts(3)} control was 1.7$\times$
slower than the production mask, contradicting Tempel's 2019
result and cancelling the conditional follow-up.

\paragraph{Validated.} EX-25 measured the parallel-walker lever
at $T \in \{1, 2, 4, 8, 14\}$. T=4 reached 2.47$\times$ the
single-threaded throughput; the system-CPU-per-thread cost
stayed flat through T=4 then jumped super-linearly. Chapter~12
gives the curve. The parallel slice landed with default
\texttt{min(hw.physicalcpu, 4)} and a synthetic-tree regression
test asserting byte-identical output across schedulers.

\paragraph{Ruled out.} Spotlight as a metadata oracle (by-design
incomplete coverage, no scoped-subtree streaming API, no
row-formatted output, undocumented store format, write
visibility lag). \texttt{getattrlistbulk} buffer size larger
than 64~KiB (the per-syscall floor is sub-0.1\% of the budget).
\texttt{getdirentriesattr(2)} (deprecated since macOS 10.10,
routes through the same VFS path on APFS).

\paragraph{Other slices shipped.} A firmlink-overlay dedup that
cut whole-machine \texttt{/} scans from 5.25M to 3.06M entries by
refusing the \texttt{/System/Volumes/Data/*} duplicate paths
(chapter~11); a progress-cadence change from one second to
250~ms so the stopwatch on a fast CLI scan does not freeze; and
the native shell developed in the next subsection, which moved
the visualisation from a WebKit-embedded JS treemap to an
in-process Rust-and-Core-Graphics renderer.

\subsection{The Native Shell}

The lane's product surface is a SwiftUI app whose visualisation
is laid out by Rust and drawn by Core Graphics. The scanner is
not a subprocess; the Rust crate is linked into the app process
as a static library, and every call from Swift into the
indexer crosses a \texttt{cbindgen}-generated C ABI. There is
no JSON serialisation, no temp file, no WebKit, and no
JavaScript.

\paragraph{Why the rewrite happened.} The pre-rewrite pipeline
emitted JSON from a subprocess, wrote it to a temp file, served
it through a \texttt{WKURLSchemeHandler}, parsed it in
WebKit, built a hierarchy in JavaScript, laid it out with
\texttt{d3.treemap}, and drew it into a \texttt{<canvas>}. A
series of optimisations --- canvas instead of SVG, bytes-aware
ingest, hot-loop rewrites of \texttt{buildHierarchy}, lazy
hit-grid construction --- pulled \texttt{/Applications}-class
scans into the one-second range. On a whole-machine scan the
JS path still spent five-to-eight seconds between the scanner
exit and the first painted rectangle. A separate probe
(documented in \texttt{docs/implementation/viz-perf-study.md})
falsified the most attractive remaining JS lever (a pure-JS
\texttt{msgpack} decoder, which V8's \texttt{JSON.parse} beat
by about three times) and projected that going native would
collapse the same window to roughly six hundred milliseconds
to one and a half seconds. The projection was the
justification for the rewrite; the rewrite landed and the
projection held.

\paragraph{The FFI boundary.} The crate's C ABI is a small
surface of opaque-handle constructors, accessors, and
destructors. \texttt{apfs\_scan\_directory} returns an
\texttt{ApfsScan*}; the handle owns the entry vector, the
indexed tree, and the strings the Swift side reads through
\texttt{String(cString:)}. \texttt{apfs\_layout\_new} returns
an \texttt{ApfsLayout*} whose flat \texttt{ApfsCell} array
Swift reads as an \texttt{UnsafeBufferPointer<ApfsCell>} ---
no per-cell call into Rust during paint, and no array copy
across the language boundary. Bulk data is handed across as
\texttt{(*const Item, usize)} pairs; per-handle \texttt{\_free}
functions are responsible for releasing every allocation
\texttt{\_new} produced. The header is regenerated by
\texttt{cbindgen} on every change to \texttt{src/ffi.rs} so the
two sides cannot drift in the type signatures they agree on.

\paragraph{The division of labour.} Rust owns: the fallback
walker, the firmlink dedup, the indexed tree (one
\texttt{TreeNode} per directory and per leaf, parent and
children indices, pre-computed \texttt{value\_logical} /
\texttt{value\_allocated} / \texttt{item\_count} aggregates),
the squarified treemap layout (a direct port of
\texttt{d3-hierarchy}'s \texttt{treemapSquarify}), the
$64 \times 64$ hit-grid, the per-extension aggregator
behind the right-hand panel, and the colour palette that
decides a leaf's \texttt{fill\_rgb}. Swift owns: the SwiftUI
shell (toolbar, breadcrumb, status bar, depth stepper, metric
picker), the AppKit treemap view (one \texttt{NSView} whose
\texttt{draw(\_:)} walks the cell array as one fillRect per
cell, grouped by colour to amortise \texttt{setFillColor}
calls), the tree-list and ext-list panels, and the right-click
menu plumbing onto \texttt{NSWorkspace} for the
\texttt{Open}, \texttt{Reveal in Finder}, \texttt{Copy Path},
and \texttt{Move to Trash} actions. The renderer touches no
JavaScript and no DOM; the shell touches no APFS bytes.

\paragraph{What got removed.} The repository still ships a
standalone \texttt{viz/index.html} for drop-a-file-into-browser
use (a separate workflow that does not require the native
build), but the app itself no longer embeds it. With the
WebKit dependency gone, several layers of intermediate
representation went with it: the JSON serialiser as a
post-scan step, the \texttt{rmp-serde} msgpack encoder, the
JavaScript \texttt{buildHierarchy} hot loop, the
\texttt{d3.treemap.squarify} call site, the \texttt{<canvas>}
DOM node, the JavaScript hit-grid, the
\texttt{WKURLSchemeHandler}-via-\texttt{ArrayBuffer} ingest
path, and the JS-to-Swift bridge protocol that carried the
typed messages between them. The scanner subprocess is gone;
so is the post-scan ingestion gap that motivated the loading
overlay an earlier slice of the lane introduced.

landed. The toolbar and the treemap content read the same in

\paragraph{Future direction.} The user-space post-processing
tax (roughly 110 thousand entries per second of budget between
the microbench kernel ceiling and the production end-to-end
throughput) is the next bounded scanner-side lever; candidates
recorded for context are \texttt{NamespaceEntry} allocation,
per-worker merge, \texttt{build\_aggregates}, and the final
sort. On the rendering side: the in-process FFI does not
surface live progress events today (the scan returns from one
synchronous call, which is fast enough on
\texttt{/Applications}-class trees but leaves the
whole-machine scan without a stopwatch), and a
threaded-scan-with-progress FFI is the natural next step. The
CLI's \texttt{--progress} path still works for headless use.

\subsection{The Treemap Layout}

The visualisation's correctness is now a Rust correctness
question. Three pieces are worth fixing in this manual; each is
small, but each is the kind of detail that decides whether two
pictures of the same data agree.

\paragraph{Squarify.} The layout is the standard squarified
treemap of Bruls, Huijzen, and van Wijk (2000), in the variant
\texttt{d3-hierarchy} implements. Children are sorted
descending by the active metric; greedy row-packing on the
shorter side of the current rect adds children to a row until
the worst-case aspect ratio would worsen, then lays the row out
by dice (horizontal) or slice (vertical) and recurses on the
remaining strip. The Rust port keeps the same
$\alpha = \max(dy/dx,\ dx/dy)\,/\,(\mathrm{value} \cdot
\mathrm{ratio})$ and $\beta = \mathrm{sum\_value}^2 \cdot
\alpha$ expressions the d3 source uses; for the same input
\texttt{(node\_index, value)} array and the same viewport, the
two implementations produce the same rectangles.

\paragraph{Hit-grid.} A $64 \times 64$ axis-aligned
spatial hash sits over the laid-out cell array. Each cell is
inserted into every bucket it overlaps; each bucket is then
sorted depth-descending so the first containing match a
hit-test walk sees is the deepest cell at that point. This is
what makes a mouse-move event a few-cell comparison rather
than a walk through the hundred-thousand-cell layout for a
\texttt{/}-scan view.

\paragraph{The unclaimed swatch.} When the metric picker is on
\texttt{Allocated} and a leaf's \texttt{value\_allocated} is
\texttt{None} (SR-019 None-collapse, chapter~8), the cell's
\texttt{fill\_rgb} is the unclaimed grey rather than a colour
from the extension palette. The rule is one branch at layout
time, but it is the visual surface of the fail-closed
discipline: the user sees, on the picture, exactly the cells
the parser refused to claim a number for.

\section{Beyond R2}

Four areas remain deliberately deferred past R2. Each is large
enough to warrant its own gate and its own oracle work; each is
unsafe to open opportunistically.

\paragraph{Encryption.} FileVault-encrypted volumes and
hardware-backed encryption have separate threat models, separate
privilege requirements, and separate validation problems. The
parser currently fails closed on encryption flags; opening
encryption support requires deciding which classes of key access
the product is willing to use and which it is not.

\paragraph{Live boot disk.} The startup disk on modern macOS is
a sealed system volume joined to a data volume by firmlinks,
with a snapshot pinning the system content. Raw scanning of the
System volume is constrained by the seal; raw scanning of the
Data volume is constrained by the live-churn problem; the
merged user-visible view depends on firmlink table semantics
that are themselves not fully public. Opening live boot support
means building each of those subsystems' oracle.

\paragraph{Boot-root merged namespace.} The Finder-visible
\texttt{/} on a modern Mac is not any single APFS volume's
namespace. It is the System volume's content joined to the
Data volume's content through firmlinks, plus the current
snapshot overlay. The product mode for this is distinct from
raw-single-volume; it cannot share the same oracle, and its
support claims must be stated separately. The firmlink-dedup
slice in chapter~11 is the closest the project has come to
this: it suppresses the on-disk duplication at the fallback
layer without claiming Finder semantics.

\paragraph{Persistent incremental cache.} The work of
chapter~10 turns into product only after the full set of
cache-invalidation conditions has been enumerated, a
crash-safe cache storage design exists, and an incremental scan
has been validated against a fresh full reparse. The R2 lanes
deliberately precede this work; a cache built on top of
incomplete metrics or unstable support matrices would propagate
its instability.

\section{The Manual's Own Future}

The chapter ends where the manual ends. Each lane above will,
when it lands, become its own chapter: a thesis, the evidence,
the precedence rule or the oracle pairing, the per-case table,
the limits of what was checked. The chapter list will lengthen;
the voice should not change.

The discipline that produced R1 produced something narrow but
defensible. The discipline that produced R2 produced something
broader, with the same defensibility intact: a metric that
landed, a lane that was correctly entitlement-gated, a lane
whose perf claim is empirically calibrated against a contention
regime the project did not invent and does not pretend does not
exist. Anything written here that survives the next year of
work survives because it was written carefully. Anything that
does not survive was an error worth recording so the next
attempt does not repeat it.
