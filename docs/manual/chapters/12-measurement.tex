\chapter{Measurement}

The project's policy on performance is straightforward: measure
before optimising; record what was measured so future claims can be
compared against it; never report a number whose source class is
ambiguous.

This chapter contains the reproducible measurements
\texttt{apfs-fastindex-scan} has produced since R1 was complete.
There are three passes: the standing baseline (chapter~11's
fallback first cut), a user-space micro-optimisation pass on the
fallback walker, and a parallel-walker pass that follows from the
performance-lever map. Each pass changes one thing at a time and
records the before/after numbers.

Reading the standing baseline:

\paragraph{The proof-fixture raw number is dominated by setup.}
Most of the 0.23~s is \texttt{hdiutil attach}; the actual
seven-entry walk is sub-millisecond. The raw-mode throughput on a
meaningful real volume is not yet measured because the project
does not carry a large detached APFS sample. The number is honest
but unrepresentative.

\paragraph{Fallback throughput sits at roughly 130 thousand
entries per second on warm trees with \texttt{lstat} and roughly
157 thousand with \texttt{getattrlistbulk}.} The improvement on
\texttt{/Applications} is about 24 per cent throughput. On the
cold \texttt{/} scan the improvement is smaller: 16 per cent
wall, because the disk is the bottleneck on a cold-cache run.

\paragraph{The cleaner perf signal is system CPU.} On the cold
\texttt{/} scan, \texttt{getattrlistbulk} drops system CPU from
48~s to 30~s --- a 38 per cent reduction. Batching cuts the
syscall count, and the kernel does less work per entry. The
wall-time savings compound on warm-cache runs where the disk is
not the constraint.

\section{The Micro-Optimisation Pass}

The standing baseline left fallback throughput on
\texttt{/Applications} at roughly 172 thousand entries per second
end-to-end (single-threaded, post-baseline) with system CPU
dominating the budget. A user-space pass against per-entry
allocations and aggregate construction landed four coupled
changes:

\begin{itemize}
  \item \texttt{BulkReader} owns one 64~KiB
    \texttt{getattrlistbulk} buffer and one output
    \texttt{Vec<BulkEntry>} across the whole walk, replacing the
    fresh \texttt{vec![0u8; 65\_536]} the per-directory
    \texttt{read\_directory\_bulk} used to allocate.
  \item \texttt{build\_aggregates} switched from
    \texttt{BTreeMap<String, BTreeMap<u64, \_{}>>} keyed by owned
    ancestor paths to \texttt{HashMap<\&str, HashMap<u64, \_{}>>}
    keyed by slices borrowed from each entry's path
    \texttt{String}. The old shape walked ancestors as a
    \texttt{Vec<String>} per file --- on a five-million-row
    \texttt{/} scan, twenty-five million owned-\texttt{String}
    allocations. The rewrite walks parents inline with
    \texttt{rfind('/')} and slices; the final sorted emission
    converts to owned \texttt{String} exactly once.
  \item \texttt{EntryKind} gained \texttt{Copy} (it is a
    four-variant unit enum), eliminating pointless
    \texttt{.clone()} calls in the per-entry walker hot path.
  \item The unconditional \texttt{relative\_str.clone()} per
    entry was collapsed: the mount-boundary branch is the only
    one that ever needed a second copy; the common path moves
    the \texttt{String} into the \texttt{NamespaceEntry}
    directly. Combined with the \texttt{Copy} change above, the
    steady-state inner loop pushes one entry with zero heap
    activity beyond the \texttt{String}/\texttt{Vec} growth
    itself.
\end{itemize}

\section{The Performance-Lever Map}

The micro-optimisation pass left the scanner at roughly 200
thousand entries per second on \texttt{/Applications} with 63 per
cent of the budget in system CPU. The remaining cost was
structurally kernel-side: user-space micro-opts had no further
room. A source review (SR-021) cross-read four parallel research
threads --- xnu's \texttt{getattrlistbulk} cost decomposition,
the Spotlight alternative, an open-source-tool survey, and the
\texttt{jwalk}/\texttt{rayon} prior art --- to produce a map of
which levers were structurally viable. The map flagged two
candidates worth measuring (parallelism and attribute-mask
shrink), two ruled-out alternatives (Spotlight, bulk buffer size),
and one worth A/B testing (\texttt{fts(3)}).

Two probes ran:

\paragraph{EX-24: attribute-mask shrink falsification.} A
single-threaded microbench compared
\texttt{getattrlistbulk} with three configurations on
\texttt{/Applications}: drec-only (no inode-required attributes),
the production mask
(\texttt{ATTR\_FILE\_TOTALSIZE | ATTR\_FILE\_ALLOCSIZE}), and
\texttt{fts(3)}. The drec-only and production rows landed within
one per cent of each other on every dimension --- wall, user CPU,
system CPU, throughput --- which falsifies SR-021's hypothesis
that the inode-attribute fetch path was a measurable lever on
modern Apple silicon. The \texttt{fts(3)} configuration was 1.7
times slower than the production mask, contradicting Tempel's
2019 result and cancelling the conditional follow-up. The pure
kernel throughput in this microbench was 312 thousand entries per
second, which gave the parallel probe a useful kernel ceiling
estimate.

\paragraph{EX-25: parallel walker.} A standalone parallel
microbench swept thread counts $T \in \{1, 2, 4, 8, 14\}$ (14
being the host's \texttt{hw.logicalcpu}). The verdict ladder
required both $T = 4$ throughput $\geq 2 \times T = 1$ throughput
\emph{and} $T = 4$ system CPU per thread $\leq 1.2 \times T = 1$
system CPU to declare scaling validated. Both gates passed.

\subsection{The Parallel Slice In Product}

The Rust slice landed a \texttt{--threads N} flag with default
\texttt{min(hw.physicalcpu, 4)}, computed via
\texttt{sysctlbyname("hw.physicalcpu")} with
\texttt{std::thread::available\_parallelism()} as the
cross-platform fallback. The walker's contract is preserved by
shape: per-worker \texttt{BulkReader}, shared
\texttt{WorkQueue} (mutex plus condvar plus outstanding counter
plus an atomic done flag), per-worker entry vectors concatenated
on the main thread, the existing
\texttt{sort\_unstable\_by} plus \texttt{build\_aggregates}
normalising ordering. A synthetic-tree regression test
(\texttt{parallel\_walker\_matches\_serial\_shape}) asserts
byte-identical entries, aggregates, and walk-skip lists across
the two schedulers.

End-to-end measurement on \texttt{/Applications}, same host as
the previous tables, median of five runs each side:

Cumulative \texttt{/Applications} throughput across both perf
passes: 172\,283 $\to$ 312\,879 entries per second, $+82\%$
($1.82\times$). The user-CPU gap between the production scanner
(256~ms at $T = 4$) and the parallel microbench (about 20~ms at
$T = 4$) quantifies the user-space post-processing tax ---
\texttt{NamespaceEntry} allocation, per-worker merge, final sort,
aggregate construction. It is recorded as a candidate future
optimisation; not load-bearing today.

\section{What The Numbers Do Not Say}

The table is a baseline; it is not a claim about anything beyond
what it measures. Three categories are explicitly outside it:

\begin{enumerate}
  \item Raw-mode throughput on a real source. The project has not
    measured raw mode on a representative live or detached
    volume of meaningful size. Until it does, the comparison
    ``raw vs fallback'' cannot be made.
  \item Incremental scans. The current model is full reparse on
    every run; the cache work of chapter~10 has not landed in
    product. Repeat-scan numbers will only mean something after
    a cache implementation passes its own oracle.
  \item Encrypted or live-boot-disk surfaces. The parallel
    walker has not been measured on encrypted APFS; per
    SR-021's review of Petrenko (2020), the surviving
    contention may behave differently there.
\end{enumerate}

\section{The Measurement Discipline}

Three habits keep the numbers honest.

\paragraph{Measure before optimising.} Each pass landed only
after a microbench established what the lever was worth. The
micro-optimisation pass was justified by an allocator-pressure
profile; the parallel pass was justified by EX-25's scaling
verdict; the attribute-mask refactor was killed by EX-24's
falsification of the vnode-rage cost.

\paragraph{Record the source class and the semantic mode with
each number.} The tables above carry both. A throughput number
without those is not interpretable: 300 thousand entries per
second is either impressive or unimpressive depending on whether
it ran cold or warm, against raw or fallback, single-threaded
or parallel, against \texttt{/Applications} or \texttt{/}.

\paragraph{Hold the baseline.} Any future performance claim has
to compare against the tables above or against a row added with
the same shape. A new optimisation that does not improve a
baseline row is not yet evidence of an improvement; it is a
hypothesis awaiting measurement. The discipline that keeps the
tables honest is the same discipline that keeps the manual
honest.
