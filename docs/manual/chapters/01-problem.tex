\chapter{Why APFS Resists MFT-Style Indexing}

WizTree's speed on NTFS comes from the Master File Table: a single
flat structure in which every file is one record. Indexing reduces
to a sequential scan. No recursion, no per-file \texttt{stat}, no
path traversal.

APFS provides no equivalent. There is no flat metadata table, by
design.

\section{What APFS Provides}

An APFS container is a copy-on-write region of transactional
B-trees. To enumerate one volume, a reader must traverse at least
three of those trees under one pinned transaction state:

\begin{itemize}
  \item a stable block-zero locator that points at the
    \emph{checkpoint descriptor area},
  \item the descriptor area, a ring of container superblocks (one
    per transaction), each a self-consistent view at that
    transaction,
  \item the \emph{container object map} (OMAP), a B-tree resolving
    virtual object identifiers to physical block addresses under a
    chosen transaction,
  \item one or more \emph{volume superblocks}, reached through the
    container OMAP,
  \item per-volume OMAPs, resolving the volume's virtual objects,
  \item per-volume \emph{file-system trees}, B-trees of
    variable-length records keyed by \texttt{(object\_id, type)}
    carrying directory entries, inodes, extended attributes,
    hard-link records, and data-stream metadata.
\end{itemize}

Every metadata read therefore costs three indirections: container,
volume, FS tree. Each indirection requires the correct OMAP and
the correct transaction context.

\section{The Right Question}

\begin{quote}
What is the minimum set of structures we must traverse, in what
order, under what state pin, to correctly enumerate one volume?
\end{quote}

\noindent The answer fills most of this manual. In compressed form:

\begin{enumerate}
  \item Locate the descriptor area from block zero.
  \item Select one checkpoint from that area --- the
    highest-XID checksum-valid container superblock.
  \item Walk the checkpoint map; resolve the container OMAP.
  \item Look up the target volume superblock through the
    container OMAP.
  \item Open the volume OMAP; look up the FS-tree root.
  \item Walk the FS tree, resolving internal-node child
    references through the volume OMAP at every level.
  \item Decode FS-tree leaf records into directory entries,
    inodes, xattrs, sibling links, and dstreams.
  \item Reconstruct paths, file identity, and size from those
    records.
\end{enumerate}

Each step is independently load-bearing. Skip the OMAP lookups at
the FS-tree internal level (chapter~\ref{ssec:fs-tree-internal})
and the walker reads random unallocated blocks. Pick the wrong
checkpoint (chapter~3) and the walker reads an incoherent
transaction. Mis-decode an xfield blob
(chapter~\ref{ssec:xfield-spec-vs-impl}) and the parser silently
emits wrong sizes.

\section{The Defended Core}

The manual's scope is deliberately narrow:

\begin{itemize}
  \item one APFS volume,
  \item at one pinned transaction state,
  \item correct namespace,
  \item per-file logical, allocated, and clone-deduplicated
    (``real'') size,
  \item fail closed outside the tested allowlist.
\end{itemize}

The scope is narrow because the cost of being approximately right
is unbounded: aggregates derived from silently wrong rows
propagate into treemaps, into ``where did my disk space go''
decisions, and into filesystem operations the user takes based on
those numbers. The cost of being narrowly right and failing closed
everywhere else is bounded: the fallback path produces a correct
answer through supported APIs, more slowly.
